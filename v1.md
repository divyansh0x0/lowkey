# Lowkey v1 — Architecture & Flow

## Overview

Lowkey is a peer-to-peer encrypted chat app. The server's only job is matchmaking — once two users are connected, all messages flow **directly between their phones** via WebRTC, encrypted with AES-256-GCM.

```
┌──────────┐         ┌──────────────┐         ┌──────────┐
│ Phone A  │◄──P2P──►│  Go Server   │◄──WS───►│ Phone B  │
│ (Alice)  │  WebRTC │  (Signaling) │  WebSocket│  (Bob)   │
└──────────┘         └──────────────┘         └──────────┘
     ▲                                              ▲
     └──────── Direct DataChannel (E2E) ────────────┘
```

> [!NOTE]
> After the WebRTC connection is established, the server is no longer involved in message delivery.

## Complete User Flow

### Phase 1: Username Setup

```mermaid
sequenceDiagram
    actor Alice
    participant App as Flutter App
    participant SP as SharedPreferences

    Alice->>App: Opens app for the first time
    App->>SP: Check for saved username
    SP-->>App: null (no username)
    App->>Alice: Show UsernameScreen
    Alice->>App: Types "alice" → taps "Get Started"
    App->>SP: Save username = "alice"
    App->>Alice: Navigate to ChatScreen
```

### Phase 2: Server Connection

```mermaid
sequenceDiagram
    participant A as Alice's App
    participant S as Go Server
    participant B as Bob's App

    A->>S: WebSocket connect: ws://IP:8080/ws?username=alice
    S->>S: hub.Register("alice", conn)
    S-->>A: Connection open ✓ (green dot)

    B->>S: WebSocket connect: ws://IP:8080/ws?username=bob
    S->>S: hub.Register("bob", conn)
    S-->>B: Connection open ✓ (green dot)
```

### Phase 3: Connect by Username

```mermaid
sequenceDiagram
    participant A as Alice's App
    participant S as Go Server
    participant B as Bob's App

    A->>S: {"type": "session:connect", "target": "bob"}
    S->>S: Create session + generate 32-byte AES key
    S->>S: Auto-join Bob to session

    S-->>A: {"type": "session:joined", "peer": "bob", "key": "base64..."}
    S-->>B: {"type": "session:joined", "peer": "alice", "key": "base64..."}
    Note over A,B: Both now have the same encryption key
```

### Phase 4: WebRTC P2P Connection

```mermaid
sequenceDiagram
    participant A as Alice (Initiator)
    participant S as Go Server (Relay only)
    participant B as Bob

    A->>A: Create RTCPeerConnection + DataChannel
    A->>A: Create SDP Offer
    A->>S: {"type": "signal:offer", "target": "bob", "payload": SDP}
    S->>B: Relay offer → Bob

    B->>B: Set remote description, create Answer
    B->>S: {"type": "signal:answer", "target": "alice", "payload": SDP}
    S->>A: Relay answer → Alice

    A->>S: {"type": "signal:ice", "target": "bob", "payload": ICE}
    S->>B: Relay ICE candidate
    B->>S: {"type": "signal:ice", "target": "alice", "payload": ICE}
    S->>A: Relay ICE candidate

    Note over A,B: 🔒 DataChannel OPEN — P2P connected!
    Note over S: Server's job is done. It never sees messages.
```

### Phase 5: Encrypted Chat

```mermaid
sequenceDiagram
    participant A as Alice
    participant B as Bob

    A->>A: Encrypt("hello") → AES-256-GCM(key, random IV)
    A->>B: Send via DataChannel: "iv:ciphertext" (base64)
    B->>B: Decrypt(ciphertext, key, iv) → "hello"
    B-->>A: Display message ✓
```

## Crypto Details (v1)

| Property | Value |
|:---------|:------|
| **Symmetric Cipher** | AES-256-GCM |
| **Key Size** | 32 bytes (256 bits) |
| **IV/Nonce** | 12 bytes, random per message |
| **Key Generation** | Server-side via Go `crypto/rand` |
| **Key Distribution** | Server → both peers over WebSocket |
| **Message Format** | `base64(IV) + ":" + base64(ciphertext + GCM tag)` |

## Known Limitation

> [!CAUTION]
> **Trust-the-server model**: The server generates and distributes the encryption key. A compromised server could retain the key. v2 will use **client-side X25519 Diffie-Hellman** (libsodium) so the server never sees the shared secret.
